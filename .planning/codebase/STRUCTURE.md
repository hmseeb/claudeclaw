# Codebase Structure

**Analysis Date:** 2026-02-26

## Directory Layout

```
claudeclaw/
├── Dockerfile              # Container image definition; clones ClaudeClaw, installs deps
├── init.sh                 # Startup script; generates config, manages state, launches daemon
├── .dockerignore           # Docker build exclusions
├── .gitattributes          # Git line-ending configuration
├── .git/                   # Repository metadata (not deployed)
└── .planning/
    └── codebase/           # GSD planning documents (architecture, conventions, etc.)
```

## Directory Purposes

**Project Root (`/`):**
- Purpose: Deployment configuration and container image definition
- Contains: Docker build artifacts, startup scripts, configuration files
- Key files: `Dockerfile`, `init.sh`
- Committed: Yes (all files)
- Deployed: All files copied into image; init.sh executed at container start

**.planning/codebase/:**
- Purpose: Architectural and quality analysis documents for GSD orchestration
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md
- Key files: This directory drives `/gsd:plan-phase` and `/gsd:execute-phase` decisions
- Committed: Yes
- Deployed: Not included in container image (development-only)

**src/ (cloned at build time):**
- Purpose: ClaudeClaw application source code (not in this repository)
- Location: Created during Docker build from `https://github.com/moazbuilds/claudeclaw.git`
- Contains: TypeScript implementation, prompts, configuration
- Key files: `src/index.ts` (daemon entry point)
- Committed: No (not in this repo)
- Deployed: Yes (cloned and installed in image at `/app/src`)

**prompts/ (cloned at build time):**
- Purpose: ClaudeClaw prompt templates and system instructions
- Location: Created during Docker build from upstream
- Committed: No (not in this repo)
- Deployed: Yes (copied to image at `/app/prompts`)

**node_modules/ (generated):**
- Purpose: Dependencies installed via Bun package manager
- Generated: Yes (`bun install --production`)
- Committed: No (excluded via .gitignore)
- Deployed: Yes (production dependencies only)

**Runtime State Directory (/app/.claude/claudeclaw/):**
- Purpose: Persistent state for daemon and session management
- Created: At container startup by init.sh
- Contains: Settings configuration, job logs, session data
- Subdirectories:
  - `jobs/`: Job queue and execution logs
  - `logs/`: Daemon and activity logs
- Deployed: On `/data` volume if provided by Railway; otherwise ephemeral

## Key File Locations

**Entry Points:**

**`Dockerfile`**
- Purpose: Container image definition
- Triggers: Docker build process
- Responsibilities:
  - Establish base image (oven/bun:1-debian)
  - Install Node.js 22 and Claude Code CLI
  - Clone upstream ClaudeClaw
  - Install production dependencies
  - Copy init.sh and mark executable
  - Set working directory to /app
- Key Commands:
  ```dockerfile
  RUN git clone --depth 1 https://github.com/moazbuilds/claudeclaw.git /tmp/claudeclaw
  COPY init.sh .
  CMD ["bash", "init.sh"]
  ```

**`init.sh`**
- Purpose: Container startup orchestration
- Triggers: Container start (CMD in Dockerfile)
- Responsibilities:
  - Create state directory structure
  - Parse environment variables
  - Generate settings.json from configuration
  - Write OAuth credentials to ~/.claude/.credentials.json
  - Establish persistent storage symlinks
  - Log initialization status
  - Launch ClaudeClaw daemon
- Key Sections:
  - Lines 1-18: Setup state directories and credentials
  - Lines 22-26: OAuth credential handling
  - Lines 28-36: Port and timezone configuration
  - Lines 38-56: Telegram, model, security, heartbeat settings
  - Lines 58-71: settings.json generation with conditional write
  - Lines 73-81: Persistent storage setup and Railway volume mounting
  - Lines 83-89: Startup status logging
  - Lines 91: Daemon launch command

**Configuration:**

**`settings.json` (generated at runtime)**
- Purpose: ClaudeClaw daemon configuration
- Generated by: `init.sh` from environment variables
- Location: `/app/.claude/claudeclaw/settings.json`
- Structure:
  ```json
  {
    "model": "opus|gpt|custom",
    "api": "api_base_url",
    "fallback": { "model": "...", "api": "..." },
    "timezone": "UTC|America/New_York|...",
    "timezoneOffsetMinutes": 0,
    "heartbeat": {
      "enabled": true,
      "interval": 30,
      "prompt": "custom_heartbeat_prompt",
      "excludeWindows": []
    },
    "telegram": {
      "token": "bot_token_from_env",
      "allowedUserIds": [123, 456, 789]
    },
    "security": {
      "level": "moderate|strict|permissive",
      "allowedTools": [],
      "disallowedTools": []
    },
    "web": {
      "enabled": true,
      "host": "0.0.0.0",
      "port": 4632
    }
  }
  ```

**`~/.claude/.credentials.json` (generated at runtime)**
- Purpose: OAuth credentials for Claude API authentication
- Generated by: `init.sh` from CLAUDE_CREDENTIALS environment variable
- Location: `$HOME/.claude/.credentials.json` (readable by daemon)
- Content: Full JSON credential object passed as env var at deployment time
- Security: Stored as file, never logged or exposed

**Build Configuration:**

**`package.json` (cloned at build time)**
- Location: `/app/package.json` (from upstream)
- Purpose: Dependency manifest for Bun
- Key fields: Scripts, dependencies, devDependencies
- Not in this repo: Cloned from moazbuilds/claudeclaw

**`tsconfig.json` (cloned at build time)**
- Location: `/app/tsconfig.json` (from upstream)
- Purpose: TypeScript compilation configuration
- Not in this repo: Cloned from moazbuilds/claudeclaw

**`.gitattributes`**
- Purpose: Enforce line endings
- Config: `*.sh` and `Dockerfile` → LF (Unix) line endings
- Why: Ensures scripts work in Linux container even if developed on Windows/macOS

**`.dockerignore`**
- Purpose: Exclude files from Docker build context
- Contents:
  ```
  node_modules      # Don't copy; bun install creates fresh
  .git              # Not needed in image
  *.log             # Temporary logs
  .env              # Secrets never committed
  .env.*            # Environment-specific secrets
  ```

## Naming Conventions

**Files:**
- Shell scripts: `*.sh` - Bash/shell script files (e.g., `init.sh`)
- Docker: `Dockerfile` - Standard Docker image definition
- Configuration: `*.json` - JSON-formatted configuration files
- Documentation: `*.md` - Markdown documentation

**Directories:**
- Source code: `src/` - Application TypeScript (cloned at build)
- Configuration: `.planning/codebase/` - Analysis documents
- System: `.git/` - Version control metadata
- Runtime: `/app/.claude/claudeclaw/` - State and logs

**Environment Variables:**
- Configuration: `CLAUDECLAW_*` - Application settings (MODEL, SECURITY)
- Infrastructure: `PORT`, `TIMEZONE` - System/platform variables
- Credentials: `CLAUDE_CREDENTIALS`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_USER_IDS` - Secret/auth data
- Flags: `HEARTBEAT_ENABLED`, `FORCE_CONFIG` - Boolean settings
- Naming pattern: `UPPERCASE_WITH_UNDERSCORES`

## Where to Add New Code

**Deployment Configuration Changes:**
- Location: `init.sh`
- Pattern: Add new environment variable → parse in shell → add field to settings.json template
- Example: Add new feature flag
  ```bash
  # 1. Add env var handling
  FEATURE_FLAG="${MY_NEW_FEATURE:-false}"

  # 2. Add to settings.json template
  "features": {
    "newFeature": ${FEATURE_FLAG}
  }
  ```

**New Service Integration:**
- Location: `init.sh` (configuration section) + upstream ClaudeClaw `src/`
- Pattern: Add service token env var → configure in settings.json → daemon loads at startup
- Example: Slack integration
  ```bash
  # In init.sh after line 47
  SLACK_TOKEN="${SLACK_BOT_TOKEN:-}"

  # In settings.json template after telegram block
  "slack": {
    "token": "${SLACK_TOKEN}",
    "enabled": $([ -n "$SLACK_TOKEN" ] && echo "true" || echo "false")
  }
  ```

**Container Base Image Changes:**
- Location: `Dockerfile` RUN commands
- Pattern: Add system package installation before ClaudeClaw clone
- Example: Add system tool
  ```dockerfile
  RUN apt-get update && apt-get install -y --no-install-recommends \
      my-new-tool \
      && apt-get clean && rm -rf /var/lib/apt/lists/*
  ```

**State Directory Structure Changes:**
- Location: `init.sh` lines 12-14
- Pattern: Add new `mkdir -p` for new state subdirectory
- Example: Add metrics directory
  ```bash
  METRICS_DIR="$STATE_DIR/metrics"
  mkdir -p "$STATE_DIR" "$JOBS_DIR" "$LOGS_DIR" "$METRICS_DIR" "$CRED_DIR"
  ```

**Application Logic (ClaudeClaw Core):**
- Location: NOT in this repo; in upstream moazbuilds/claudeclaw
- Pattern: Fork/patch upstream or submit PR
- Note: This deployment wrapper assumes upstream ClaudeClaw is read-only; logic lives there

## Special Directories

**`.planning/codebase/`:**
- Purpose: GSD (Generalist Self-Directed) orchestration documents
- Generated: No (hand-written analysis)
- Committed: Yes
- Contents:
  - `ARCHITECTURE.md` - Layer structure, data flow, abstractions
  - `STRUCTURE.md` - Directory layout, file locations, naming patterns
  - `CONVENTIONS.md` - Code style, naming patterns, error handling
  - `TESTING.md` - Test framework, patterns, coverage
  - `CONCERNS.md` - Technical debt, fragile areas, scaling limits
- Usage: Loaded by `/gsd:plan-phase` and `/gsd:execute-phase` to understand codebase conventions and structure
- Note: Not included in Docker image; development-only

**`/app/.claude/claudeclaw/` (Runtime)**
- Purpose: ClaudeClaw daemon state
- Generated: Yes (created by init.sh at container startup)
- Committed: No (ephemeral or volume-persisted)
- Subdirectories:
  - `jobs/` - Job queue and execution logs
  - `logs/` - Daemon activity and error logs
- Volume Mounting: If Railway provides `/data`, this directory is symlinked to `/data/claudeclaw` for persistence across container restarts

**`/data/` (Optional Railway Volume)**
- Purpose: Persistent storage for logs and session state
- Generated: No (provided by Railway infrastructure)
- Committed: N/A (external)
- Relationship: `init.sh` symlinks `/app/.claude/claudeclaw` → `/data/claudeclaw` if `/data` exists
- Effect: Logs and job state survive container restart
